HLAPI Structure

High Level API Documentation

This section goes into detail regarding the detailed implementations of the HLAPI

Introduction

As indicated in Getting Started, the first part of this section is a reiteration of that which was described. In subsequent parts, detailed discussions will be carried out.s

  • The AAMotion HLAPI talks with the motion controllers via a communication server. This server can be connected by multiple clients.

    This means that PCSuite; an application software which provides a front end GUI which can be used to configure a motion controller can also inter-operate with the AAMotion simultaneously

    The communication Server (also known as CommServer) handles all the incoming and outgoing traffic to any number of controllers

    In fact there can be multiple instances of HLAPI and PCSuite and it will all still work transparently with each other. A detailed discussion of this can also be found on the remarks section of MotionController

  • Once AAMotion DLL has been installed at programmer's operating environment, a simple Connect(String) is necessary to establish connection between user and motion controller. Thereafter, the user can use any of the Nodes (as described in more detail in HLAPI Structure). To control either a single axis motion, a vector/multiple axis motion or an IO pertaining to the relevant port that it is connected to.

  • After instantiating a MotionController class, the user will need Connect(String) to the motion controller and verify that connection is successful (IsConnected) is true), thereafter user will be able to actuate on all the INode that the controller manages, such as a ISingleAxisINode like AxisA/AxisB/AxisC or the vector/multiple axis node Group or any of the IIOModule associated with AxisA/AxisB/AxisC, IO_A/IO_B/IO_C

The Notion of Node

The idea of a node is a component that pertains to either a Single Axis, Group Axis or IOModule. Or basically anything which may contain properties (described in next section) such as status or settings. basically the notion of a Node is a component which contains a state.

  • A Node is described in detail in the remarks of AAMotion

Properties, PDOs and SDOs

Properties, PDOs and SDOs

  • Properties are used to describe any statuses or settings pertaining to a INode.

  • The AAMotion HLAPI essentially has a scheduler residing within each INode which is periodically querying from the motion controller via CommServer (using LLAPI). And the property will be queried if it is configured as a PDO by using ConfigurePDO. The rate at which a property is configured depends on its priority level. In general Statuses have a higher priority than Settings.

  • Similarly a Status or Setting Property can be configured as an SDO meaning that when a user queries for this value, the call to get the property itself is actually a sequence of instructions to poll an instruction into the CommServer to request for the designed property. The method to call is ConfigureSDO

  • A Status is a property that can change as a result of external changes in the environment or as a factor of the motion controller itself. Such as ActualPosition

  • A Setting is a property that is generally set by the user. It is necessary for the nodes within a MotionController of AAMotion HLAPI to query these properties since Application Layer instances may talk to LLAPI and configure the motion controller independently of the HLAPI instance

Method Calls

A Method calls is a function called within a class. In AAMotion HLAPI, all method calls return as boolean type.

The return of each method call therefore indicates whether calling this method was successful or not

If a method call is successful then thats that, the user can proceed to use it as how it was intended. However if the method call return false, then must be assumed that this method call failed to execute properly, and so the user can query the ErrorMessage or ErrorCode. To see what might have cause the method call to fail.

Errors and Error Handling

All INode instances have ErrorMessage or ErrorCode property that is updated whenever a method call fails to execute properly as described above.
