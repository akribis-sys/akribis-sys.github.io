Welcome to the Technical Specifications for AAMotion High Level Application Programming Interface(HLAPI)

The following sections and links contain help and technical documentation of AAMotion HLAPI for the use of controller Akribis-Agito controllers to implement motion related actuations which may be tied to user process or system environment

  • The documentation is broken up into three sections, a general user guide is described in the sections denoted under Getting Started

  • While the section denoted under API Overview is the API structure pertaining to its classes with detailed information regarding implementation, usage of methods and properties and general behavior of the HLAPI itself

  • The last section is regarding version of the HLAPI, such as release notes and main changes in a particular version as well as a development road-map of all items that have been planned and scheduled for in the development pipeline. This is presented here so that programmers and users of the HLAPI can check if there is a particular function that is not currently available and to determine whether it will be available in the near or coming future. If the situation is that there is not mention of said item, then it is recommended for the HLAPI user to email for support regarding such matters

Glossary

This section describes commonly used terms in this documentation to clarify the meaning of some specific terms

Terms

Acronym

Name

Meaning

HLAPI

High Level API

Application programming layer that is designed for user for front end usage of motion controller, such as direct actuation of motion on the Motion Controller

LLAPI

Low Level API

Application programming layer that directly communicates with the CommServer process, more details can be found in remarks of AAMotion

CommServer

Communication Server Process

This is a process that runs as a server that listens to commands issued by the LLAPI. The LLAPI can be instantiated by any number of application layer processes such as the PCSuite Application Suite or the AAMotion HLAPI

DLL

Dynamic Link Loader

This is just a standard term which denotes a library that is loaded at runtime. When AAMotion is mentioned as a entity of HLAPI, it denotes the usage of a DLL

PDO

Process Data Object

This terminology is first introduced in the CAN Open protocol and later incorporated in EtherCat as a means to indicate information which is periodically synchronized between controller and driver. For documentation and explanation purposes, this term is also used as a means to describe how a user can access status and settings properties pertaining to the motion controller or to a node within the motion controller

SDO

Service Data Object

This terminology is first introduced in the CAN Open protocol and later incorporated in EtherCat as a means to indicate information which is periodically synchronized between controller and driver. For documentation and explanation purposes, this term is also used as a means to describe how a user can access status and settings properties pertaining to the motion controller or to a node within the motion controller

Profiler

Profiler

In the context of the AAMotion API, the profiler is a terminology used to define a means to measure the performance of the AAMotion HLAPI. However in the context of a motion axis, this has a very different meaning, being that the profiler denotes the mechanism which generates the actuation to reach a certain target, this is irrespective of the operation mode, whether it is in Position/Velocity/Torque mode
