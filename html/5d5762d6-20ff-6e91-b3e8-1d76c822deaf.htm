Remarks
The image below depicts how PDO and SDO is realized in terms of timing and a general idea for how the scheduling works with respect to other INode in MotionController class.

Essentially for every node there are statuses and settings pertaining to that node, so AAMotion implements a way to get this information from controller in a transparent manner, by which to get these statuses and setting (called properties in the context of the documentation and programming), so what user gets (when a property is configured as a PDO) when a property is inquired, is a mirror value of the last gotten status/setting. This is achieved via the means of three internal heartbeats which fire off periodically. In this documentation, this heartbeat will also be known as an interrupt. There are three different interrupt governed by different priorities and each priority dictates a different periodic interval.

The interrupts cannot preempt each other (as of version 2.0.0.0 but may be improved in the future) so once an interrupt of either low/medium/high priority has started running, then it will continue to run until it has completely finished. When a interrupt is fired, it checks whether the communication channel is currently running and if it isn't, then the priority interrupt will start it cycle. A interrupt is tied in conjunction to fulfill the functional needs of PDO; PDO stands for Process Data Object which is a terminology borrowed from EtherCat Protocol. Basically it means synchronous mechanism to grab and synchronize information/properties (or better understood as statuses and setting of the controller) between AAMotion HLAPI. So users do not need worry about the issue of polling too frequently (if the method of getting the property is synchronous/blocking) or having to implement additional logic to call asynchronous command to get the property and then to periodically check if the command has been successfully executed by CommServer to Controller and correspondingly receiving a result.

Finally AAMotion HLAPI also facilitates the capability to disable a property from being periodically updated (which may be useful for optimization purposes). And at the same time, allows for a property to be queried immediately (meaning to be able to cut the AAMotion HLAPI traffic management of all the pending commands in the queue waiting to be executed) via the configuration of that property as a SDO. SDO means Service Data Object which is also a terminology borrowed from EtherCat. SDO basically means a mechanism to query for a property in the controller asynchronously/on-demand/as and when needed.
