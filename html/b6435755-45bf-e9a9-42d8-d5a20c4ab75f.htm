AAMotion Namespace

High Level API Documentation
The AAMotion Namespace contains High Level API access to AAMotion Controllers. There is only one entry via the MotionController Class: MotionController
Classes
  ClassDescription
MotionController
This is the entry class of HLAPI to access AAMotion Controllers, There are three basic components that MotionController motion controller may have; ISingleAxis, IIOModule and IGroupAxis. These interfaces act as the pathway to access these components. They can all be consider a Node, INode

Remarks
The image below depicts the HLAPI Structure:
MotionController is the container for all INode within a controller. A INode is a set of functions pertaining to a particular axis or port of the controller depending on which type of controller it is connected to. AxisX where X can be A/B/C relates to a single axis motion where the motor type MotorType defines what kind of motor it is. IGroupAxis pertains to a virtual axis which can be composed of any combination of A/B/C. IIOModule Pertains to the IO functions relating to each axis/port. For AG300, IO_A and IO_B is actually the same since AG300 is a dual axis integrated motion controller and there is only one IO module to it. LLAPI stands for Low Level API which is the API which allows for direct access to the communicator, it is the exact same mechanism that PCSuite is using to talk to the controllers, and in fact is the same as using the terminal to type in keywords.


The image below depicts the HLAPI Layout with respect to other processes in the same ecosystem:
For every Instance of HLAPI or PCSuite that is created, a new LLAPI instance is also create. The LLAPI functions to talk with the Communicator Process. The communicator process or CommServer is a server which talks directly to the controller ControllerType, it liaises any information between user application to the controller and acts as a traffic management to manage between multiple connections and to ensure no conflict in information is being sent at any one time.
